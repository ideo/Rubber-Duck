# Bambu Duck — Claude Code Context

You are working inside the **Bambu Duck** subtree of the Duck Duck Duck
repo. This is a sibling product to the Mac/Claude Code duck described in
the parent `CLAUDE.md`. **The two share a repo and a voice — but no
runtime overlap.** Don't conflate them.

## What this product is

A physical rubber duck that lives next to a Bambu 3D printer. ESP32-S3
on board (mic + speaker + servo + WiFi). Runs standalone — no Mac, no
plugin, no companion app. The user talks to the duck, the duck has a
real conversation through ElevenLabs Conversational AI (ElevenAgents),
and the agent can query the printer's live state mid-sentence via
relay-hosted webhook tools. The relay also pushes printer-event
notifications back to the duck so it speaks up unprompted ("Work
Bambu just finished printing the dragon").

## Three-tier architecture

```
[duck ESP32-S3]  ──wss──→  [relay (Fly.io)]  ──wss──→  [ElevenAgents]
       ↑                          │
       └────── speaker ────────── │ ──MQTT──→ [Bambu cloud broker] → [printer]
```

- **`firmware/`** — ESP-IDF v5.3.4 project. Two hardware variants:
  ducky PCB (custom WROOM-1, full-duplex I2S) and XIAO Seeed S3 +
  cobbled parts. Single source tree, build flag `DUCK_VARIANT=XIAO`
  flips pin maps. Captive-portal onboarding (APSTA WiFi wizard +
  Bambu cloud login + printer picker).
- **`relay/`** — Python FastAPI service deployed to Fly.io
  (`duck-duck-print.fly.dev`). Multi-tenant: SQLite per-duck rows
  keyed by chip MAC, separate Bambu MQTT subscription per duck,
  shared agent on ElevenAgents. Tool calls route to the right
  tenant via the `X-Duck-Id` request header (see "Multi-tenancy"
  below — there's a real bug story here).
- **`elevenlabs/agent-template.json`** — Source-of-truth for the
  ElevenAgents agent config (system prompt, tool schemas, voice).
  Templated; `bambu/DEPLOY.md` substitutes `{{RELAY_URL}}` and
  `{{RELAY_SHARED_SECRET}}` and POSTs to ElevenLabs to create a
  fresh agent per self-hoster.

## Key files (firmware)

- `firmware/main/main.c` — boot + main loop. Routes wake triggers:
  WAKE_TAP (double-tap) → conversation, WAKE_BUTTON (short press)
  → cycle volume, WAKE_LONG_PRESS → re-onboard wizard.
- `firmware/main/agent.c` — session WS to relay (`/ws/duck`),
  notify channel WS (`/ws/notify`), mic capture + speaker drain
  tasks, on_text/on_binary handlers. **`type_field_equals` is the
  whitespace-tolerant JSON match** that replaced literal substring
  matches — see commit `6fdbfa4` for the months-old mic-gate bug
  that motivated it.
- `firmware/main/audio.c` — full-duplex I2S, the chirp synth
  (sawtooth + Chamberlin SVF bandpass — ported from the original
  Teensy duck's ChirpSynth.ino), volume cycle (NVS-persisted),
  mic auto-DC tracker.
- `firmware/main/wake.c` — tap-to-wake detector with double-tap
  accumulator (cooldown 80ms, window 750ms).
- `firmware/main/provision.c` — APSTA captive portal, embedded
  HTML form, picker page, fast-path for re-onboard.
- `firmware/main/phrases.c` + `phrases/*.opus` — pre-recorded
  embedded phrases (`tap_to_start`, `wifi_up`, `wifi_connected`).
  Generated by `firmware/scripts/gen_phrases.py` via ElevenLabs
  TTS API.
- `firmware/main/servo.c` — animation, idle-hop taper (alert →
  settling → drowsy → dormant based on time since last user
  interaction).
- `firmware/main/wifi.c` — STA connection, NVS creds, SNTP for
  wall-clock time.

## Key files (relay)

- `relay/main.py` — FastAPI app, tool endpoints (path-scoped +
  un-scoped variants), admin endpoints, MQTT BambuState wiring
  per duck.
- `relay/duck_proxy.py` — `/ws/duck` (session bridge to
  ElevenAgents), `/ws/notify` (long-lived push channel), agent
  init payload with `dynamic_variables: {duck_id}` per session.
- `relay/bambu_state.py` — paho-mqtt client, push_status
  accumulation, transition detection (start/finish/failed/pause/
  resume/hms), print history with proper duration tracking and
  FINISH-before-subscribe race recovery.
- `relay/db.py` — SQLite schema + queries. Keyed by `duck_id`
  (chip's SoftAP MAC, 12 hex chars).
- `relay/bambu_cloud.py` — Bambu account login + list_devices via
  httpx, used during captive-portal onboarding.

## Multi-tenancy (read this before touching tool routing)

**The bug**: agent tool URLs are static at agent-create time. If they
point at un-scoped paths (`/tools/printer_state`), the relay falls
back to `default_duck_id()` (oldest row in DB) for every conversation
— meaning every duck on the relay reports the operator's printer
state, not the speaking user's.

**The fix** (commit `0341fe6`): tool URLs stay un-scoped, but the
agent template includes an `X-Duck-Id` request header sourced from
ElevenAgents `dynamic_variables.duck_id`. The relay's `_send_init`
passes `duck_id` per-session in `conversation_initiation_client_data`.
Tool endpoints prefer the header, fall back to default-duck only
when the header is absent (single-duck self-hoster ergonomics).

If you change anything about how duck_id flows from the WS handshake
to the agent's tool calls, **test with two ducks on one relay** —
single-duck testing won't catch a regression here.

## Build flavors

- **Default (open-source)** — captive portal exposes ElevenLabs API
  key + agent ID fields so each self-hoster brings their own account.
- **Turnkey** (`make flash-turnkey` / `idf.py -DBAMBU_DUCK_TURNKEY=1`)
  — strips those fields. For ducks you flash and hand to people on
  a relay where the ElevenLabs creds are Fly secrets.

`bambu/firmware/Makefile` has the canonical targets:
- `make build-ducky / flash-ducky` — default ducky PCB
- `make build-xiao / flash-xiao` — XIAO Seeed S3 variant
- `make build-turnkey / flash-turnkey` — ducky PCB + turnkey

## Style notes

- The duck has personality — helpful, slightly snarky, occasionally
  sentimental about prints. Same voice family as the parent project.
- Agent voice: `ygoBNrnmTEdu5NtDTmAY` (ElevenLabs). Keep this in sync
  across `elevenlabs/agent-template.json`, `firmware/scripts/gen_phrases.py`,
  and any per-duck overrides — embedded phrases must sound like the
  live agent.
- Chirps go through one synth (`chirp_squelch_note`) — sawtooth +
  bandpass — so all UI feedback shares the same squelchy character.
  Bends in the duck's vocal register (~280–680 Hz). Errors have their
  own family: `audio_chirp_uh_oh` (chip-internal) vs `audio_chirp_uh_uh`
  (printer-side fault).

## Safety rules

- **NEVER kill the user's serial monitor.** Arduino IDE serial
  monitor (or any other dev tool) holding `/dev/cu.usbmodem*` is
  off-limits. If `lsof` shows it held, **ask the user to close it
  themselves** — do not `kill`, `pkill`, or otherwise touch the
  process.
- **NEVER use `pkill -f`.** Same reason as the parent project's
  CLAUDE.md — matches full command lines and can hit system processes.
- **CAD files always commit.** When `hardware/CAD/**/*.3mf` shows
  modified, include in commits without asking.
- **Never embed secrets in code.** ElevenLabs API key and Fly shared
  secret live in Fly secrets and `.env` files (`.gitignore`d). Never
  paste them into committed source.
- **TLS verification is on by default for cloud MQTT.** The
  `verify_tls=False` LAN fallback in `bambu_state.py` exists for
  local-dev only; new BambuState construction defaults to True so
  callers fail-secure.

## Dev workflow

```bash
# Firmware
cd bambu/firmware
source ~/esp/esp-idf/export.sh
make flash-turnkey PORT=/dev/cu.usbmodem101
# Then watch:
idf.py -B build_turnkey -p /dev/cu.usbmodem101 monitor

# Relay (local dev)
cd bambu/relay
.venv/bin/uvicorn main:app --reload --port 8088

# Relay (deploy to Fly)
cd bambu/relay
fly deploy
```

Test loop pattern: edit → build → flash → monitor a focused grep of
the serial output. The chip's logs are noisy by default; filter to
the events you care about (`mic:`, `wake:`, `agent: ws`, `session
ready`, `DOUBLE-TAP`).

## Connecting to the deployed relay

`https://duck-duck-print.fly.dev` is the production deployment.
Per-duck rows live in SQLite at `/data/ducks.db` on the Fly volume,
replicated to S3 via Litestream when configured.

```bash
fly logs -a duck-duck-print              # tail logs
fly ssh console -a duck-duck-print       # shell into the VM
fly secrets list -a duck-duck-print      # see configured secrets
```

## What's intentionally NOT here

- **Destructive printer tools** (pause, stop, parameter changes).
  The agent is read-only. If we ever add an actuating tool, the
  authority model needs to land first (formerly tracked as #25,
  closed as won't-do-yet).
- **Persistent print history across relay restarts.** Held in
  memory, rebuilt on MQTT reconnect (closed #23). The duck doesn't
  need to memory-manage; the relay re-reads live state.
- **Mac-companion features.** This product doesn't talk to Claude
  Code, doesn't score prompts, doesn't read the user's screen.
  Anything related to that is the parent project's domain.
